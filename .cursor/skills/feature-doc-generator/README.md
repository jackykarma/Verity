# Feature Documentation Generator

自动为业务功能生成结构化项目文档的 Cursor Agent Skill。

## 功能概述

这个 skill 可以帮助你：

- 📄 **生成业务文档**：面向产品经理、测试人员和新入职工程师的业务级文档
- 🔧 **生成技术文档**：面向工程师的详细技术实现文档
- 🔄 **增量更新文档**：当代码变化时，智能更新现有文档
- ✅ **基于事实**：所有内容来源于代码，明确标注信息来源

## 快速开始

### 基本用法

在 Cursor 中直接对 Agent 说：

```
创建搜索功能的文档
```

Agent 会自动：
1. 探索搜索相关的代码
2. 生成业务文档和技术实现文档
3. 保存到 `docs/features/search/` 目录

### 只生成某一类文档

```
只创建用户认证功能的业务文档
```

或者

```
只创建支付功能的技术文档
```

### 更新现有文档

```
更新订单功能的文档，代码有变化
```

## 输出结构

生成的文档保存在项目的 `docs/features/` 目录：

```
docs/
└── features/
    └── [功能名称]/
        ├── business.md       # 业务文档
        └── technical.md      # 技术实现文档
```

## 文档特点

### 业务文档

- ✅ 不包含类名、方法名等技术细节
- ✅ 使用业务术语描述流程
- ✅ 包含业务规则和状态流转
- ✅ 新人可以快速理解业务逻辑

### 技术文档

- ✅ 完整的类图和时序图
- ✅ 详细的数据模型说明
- ✅ 设计决策和原因
- ✅ 扩展点和技术债说明

## 信息来源标注

所有文档都会标注信息来源：

- **✅ 已确认信息**：直接来自代码、配置文件
- **⚠️ 合理推断**：基于代码逻辑推断的内容

## 使用场景

### 1. 新功能上线

```
我们刚完成了消息推送功能，帮我生成文档
```

### 2. 新人入职

```
为视频播放模块生成业务文档，新同事需要了解
```

### 3. 代码审查前

```
生成权限系统的技术文档，准备做架构 Review
```

### 4. 补齐历史欠账

```
我们的支付模块一直没有文档，帮我补上
```

## 核心原则

### 基于事实，而非臆测

所有内容必须来源于代码、注释、配置，不凭经验补全业务。

### 明确区分信息确定性

对于不确定的内容，使用 ⚠️ 标注并说明推断依据。

### 新人优先

默认读者是第一次接触该模块的工程师，不使用隐含知识。

## 质量保证

生成的文档会自动进行质量检查：

- [ ] 业务文档不包含技术术语
- [ ] 技术文档包含完整的类图和时序图
- [ ] 所有业务规则都有代码依据
- [ ] 不确定的信息已标注来源

## 注意事项

1. **代码探索范围**：Skill 会自动识别功能边界，但大型模块可能需要你明确指定范围
2. **图表生成**：使用 Mermaid 语法生成类图和时序图，确保在 Markdown 阅读器中正确渲染
3. **增量更新**：如果文档已存在，Skill 会智能对比变化并增量更新

## 反馈与改进

如果生成的文档有问题，可以：

```
这个业务规则不对，应该是...
```

或者

```
技术文档中缺少了XX模块的说明
```

Agent 会根据你的反馈调整文档。
